Open LLM VTuber is a modular, Python-based application built on FastAPI. Each AI subsystem — LLM, ASR, TTS, and VAD — follows the same factory pattern, making it straightforward to swap implementations or add new ones.

This section covers everything you need to contribute to the project or extend it with custom components.

## Prerequisites

Before starting, make sure you have:

- Python 3.10–3.12
- [uv](https://docs.astral.sh/uv/) (modern Python package manager)
- Git

## Architecture overview

The system is built around three layers:

<CardGroup cols={2}>
  <Card title="WebSocket server" icon="plug">
    A FastAPI server (`server.py`) that accepts WebSocket connections from the browser frontend or desktop client. Each connection gets its own isolated `ServiceContext`.
  </Card>
  <Card title="Service context" icon="box">
    A dependency injection container (`service_context.py`) that holds references to the active ASR, TTS, agent, VAD, and translation engines for a single session.
  </Card>
  <Card title="Engine factories" icon="gear">
    `ASRFactory`, `TTSFactory`, and `AgentFactory` read the YAML configuration and instantiate the correct engine implementation. Adding a new provider means adding a branch to the factory.
  </Card>
  <Card title="Conversation pipeline" icon="message">
    The `conversations/` module orchestrates the full turn: audio → ASR → agent → TTS → audio → Live2D expression. Group and individual conversations are handled separately.
  </Card>
</CardGroup>

## Key development patterns

### Factory pattern for engines

Every AI engine type has a factory class and an abstract interface:

| Interface | Factory | Directory |
|---|---|---|
| `ASRInterface` | `ASRFactory` | `src/open_llm_vtuber/asr/` |
| `TTSInterface` | `TTSFactory` | `src/open_llm_vtuber/tts/` |
| `AgentInterface` | `AgentFactory` | `src/open_llm_vtuber/agent/` |
| `StatelessLLMInterface` | `LLMFactory` | `src/open_llm_vtuber/agent/stateless_llm/` |

To add a new provider, implement the interface and register the new class in the factory.

### WebSocket message handling

Incoming messages are dispatched by type through a dictionary of handlers initialised in `_init_message_handlers()` inside `WebSocketHandler`. To add a new message type:

1. Add the type string to the `MessageType` enum.
2. Write a `_handle_*` method.
3. Register it in `_init_message_handlers()`.

### Configuration management

All configuration is type-safe using Pydantic models in `config_manager/`. When you add a new engine option:

- Add a config class in `config_manager/`.
- Update `config_templates/conf.default.yaml`.
- Update `config_templates/conf.ZH.default.yaml`.
- Maintain backward compatibility wherever possible.
